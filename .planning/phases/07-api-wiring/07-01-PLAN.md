# Phase 07 Plan 01: API Path Alignment + User Provisioning

## Objective

Fix the four integration gaps that prevent the frontend from communicating with the backend:
1. API route paths mismatch (frontend calls `/api/chat`, backend serves `/chat`)
2. userId identity mismatch (frontend sends wallet address, backend expects UUID)
3. Verify-proof BFF calls wrong backend path (`/api/verify` instead of `/verify`)
4. Protected layout redirect is commented out (unauthenticated users access chat page)

## Context

- Backend: Hono app at `apps/api/src/index.ts` mounts `chatRoute`, `verifyRoute`, `confirmRoute` all under `/`
- Frontend: ChatInterface sends to `${API_URL}/api/chat` (has `/api` prefix)
- Auth: NextAuth session.user.id = wallet address (0x hex string), not a DB UUID
- DB: `users.id` is UUID, `users.walletAddress` is unique text — need upsert by wallet address to get UUID
- verify-proof BFF (apps/web/src/app/api/verify-proof/route.ts) calls `${apiUrl}/api/verify` — backend is at `/verify`

## Tasks

### Task 2: User provisioning — upsert user by wallet address in chat route
type="auto"

**What:** The chat route receives `userId` from the frontend, but the frontend's `session.user.id` is a wallet address (0x hex string). The backend currently queries `users.id` (UUID) with this wallet address and returns a stub context. 

Fix: In the chat route, detect if the provided `userId` looks like a wallet address (starts with `0x`). If so, upsert the user by wallet address and resolve the internal UUID.

**Files to modify:**
- `apps/api/src/routes/chat.ts` — add wallet-address detection + upsert logic before `fetchUserContext`

**Implementation detail:**
```typescript
// In chatRoute.post('/api/chat', ...) handler, before fetchUserContext:
async function resolveUserId(rawId: string | undefined): Promise<string | undefined> {
  if (!rawId) return undefined;
  // If already looks like a UUID, use as-is
  if (/^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i.test(rawId)) {
    return rawId;
  }
  // Treat as wallet address — upsert and return UUID
  if (rawId.startsWith('0x')) {
    const walletAddress = rawId.toLowerCase();
    // Try to find existing user
    const [existing] = await db.select({ id: users.id })
      .from(users)
      .where(eq(users.walletAddress, walletAddress))
      .limit(1);
    if (existing) return existing.id;
    // Provision new user
    const [newUser] = await db.insert(users)
      .values({
        walletAddress,
        displayName: walletAddress.slice(0, 8),  // short address as default display name
        autoApproveUsd: '25',
      })
      .returning({ id: users.id });
    return newUser?.id;
  }
  return rawId;
}
```

**Verification:** 
- Send POST /api/chat with `userId: "0x1234...abcd"` — no 500 error, user context loads
- Subsequent request with same wallet address returns same UUID (idempotent)

**Done criteria:**
- `resolveUserId` function exists in `apps/api/src/routes/chat.ts`
- chatRoute calls `resolveUserId(body.userId)` before `fetchUserContext`
- Wallet address provisioning creates a DB user and returns UUID
- UUID passthrough still works for direct UUID callers

### Task 3: Fix verify-proof BFF path
type="auto"

**What:** `apps/web/src/app/api/verify-proof/route.ts` calls `${apiUrl}/api/verify` — but with the path fix from Task 1 this is now correct. However, the BFF also sends an incomplete payload — it only sends `nullifier_hash` and `userId`, but the backend `proofSchema` requires `proof`, `merkle_root`, `nullifier_hash`, `verification_level` as well.

Fix the BFF to send the full proof payload that the backend expects.

**Files to modify:**
- `apps/web/src/app/api/verify-proof/route.ts` — fix the backend call to send full proof fields

**Current issue:**
```typescript
// BROKEN: only sends nullifier_hash + userId
await fetch(`${apiUrl}/api/verify`, {
  body: JSON.stringify({
    nullifier_hash: nullifier_hash,
    userId: userId,
  }),
});
```

**Fix — send full proof payload:**
```typescript
// After Task 1, route is /api/verify. Send complete proof fields:
await fetch(`${apiUrl}/api/verify`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    userId,
    proof: payload.proof,
    merkle_root: payload.merkle_root,
    nullifier_hash: payload.nullifier_hash,
    verification_level: payload.verification_level ?? 'orb',
  }),
});
```

Note: `payload` is the World ID proof object received from MiniKit — it contains all the required fields.

**Done criteria:**
- BFF sends full proof payload (`proof`, `merkle_root`, `nullifier_hash`, `verification_level`)
- BFF calls `/api/verify` (matches Task 1 path)
- `userId` is resolved — use `session?.user?.id` which is the wallet address; backend verify route needs same wallet-to-UUID resolution
